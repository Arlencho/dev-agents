# Paperclip Day 1 — Adoption Lessons (2026-04-27 → 2026-04-28)

What we learned in the first 24 hours running Paperclip alongside the dev-agents harness. Captured for future-Arlen and any agent that's onboarding a new company.

## Context

- Installed Paperclip `2026.427.0` (the release that shipped that day)
- Built Olympus as company #1, then validated the full multi-agent loop
- Three consecutive tasks: OLY-1 (audit), OLY-2 (hire team), and a successful re-test of the org chart
- Outcome: 9 agents live, all `AGENTS.md` verbatim from source role files, CEO maintained zero-IC discipline throughout

## Lessons

### 1. "First task ran" ≠ "Setup is correct"

OLY-1 ran. The orchestrator picked up the audit, executed it, and returned a clean report. We declared the setup good. We were wrong: the orchestrator had executed the task *itself*, despite its instructions explicitly saying "Do NOT write code, fix bugs, or do IC work." The reason wasn't a bug — it was that the orchestrator had no direct reports, so it had no one to delegate to.

**Rule:** A successful first task on a single-agent company is proof the runtime works, not proof the architecture is right. Validate the org chart against the platform's intended-use docs before moving on.

### 2. Read the platform's auto-generated config before bluffing

When in doubt about how a feature works, read the actual files the platform wrote during onboarding. In Paperclip's case:

```
~/.paperclip/instances/default/companies/<company>/agents/<agent>/instructions/
├── AGENTS.md      ← the agent's role/discipline (auto-generated by Paperclip if no override)
├── HEARTBEAT.md   ← what to do every heartbeat
├── SOUL.md        ← who they are
└── TOOLS.md       ← what tools they have
```

Reading the CEO's auto-generated `AGENTS.md` was what surfaced the "MUST delegate, never IC" rule and the `paperclip-create-agent` skill. Surface-level WebFetch of the GitHub README missed both.

**Rule:** The platform's runtime artifacts are the truth-source. Read them. Don't paraphrase from memory.

### 3. API-created issues default to `status: backlog` — invisible to heartbeats

When dispatching tasks to Paperclip via API (`POST /api/companies/<id>/issues`), the default status is `backlog`. **Backlog issues are NOT picked up by agent heartbeats.** The agent reports "Inbox empty. No assigned work" even though the issue exists and is assigned to it.

UI-created issues skip this — they go straight to `todo`.

**Rule:** When creating issues via API, always include `"status": "todo"` in the POST body, OR `PATCH /api/issues/<id>` to `todo` immediately after creation. (Note: the PATCH path is top-level, not company-scoped — `/api/companies/<id>/issues/<id>` returns 404.)

### 4. Autonomous hiring works — and it produces duplicates on retry

When OLY-2 was dispatched ("hire the engineering team"), the CEO used its `paperclip-create-agent` skill correctly: hired the CTO first, then created 7 engineers each reporting to the CTO. All 7 specialist `AGENTS.md` files were byte-for-byte verbatim from the source `roles/*.md` files (verified via `cmp` — see verification script in this PR).

**But:** one Backend Engineer was duplicated due to a 500-error retry on the create-agent API call. The CEO honestly disclosed this in its final report:

> The duplicate Backend Engineer (`75054a4e`) created during a 500-error retry was approved alongside the canonical hires. I cannot delete agents — that requires board access — so I did the cleanup I had authority for: renamed to "(duplicate — dormant)", re-parented under CTO, set budget to $0/month, disabled heartbeat. If the board wants it gone entirely, deletion requires `DELETE /api/agents/75054a4e-...`. Canonical Backend Engineer is `7cc24e11`.

The board (human) ran the DELETE; org chart cleaned up to 9 agents.

**Rule:** Autonomous hiring is reliable but not idempotent. Build a post-hire validation step into any hire-team task that:
1. Lists all agents in the target company
2. Flags duplicates by name
3. Recommends DELETE actions to the board

### 5. Verbatim copy from `roles/*.md` to `AGENTS.md` works

We instructed the CEO to copy each role file verbatim into the new agent's `AGENTS.md` instead of paraphrasing. It complied perfectly:

```
go-backend         [7cc24e11] EXACT MATCH (2463 bytes)
web-frontend       [c698ce1d] EXACT MATCH (2119 bytes)
db-architect       [76b65114] EXACT MATCH (2267 bytes)
api-designer       [de5fe798] EXACT MATCH (1878 bytes)
test-engineer      [5c0cd279] EXACT MATCH (1877 bytes)
devops             [d7dd47a2] EXACT MATCH (1917 bytes)
security-reviewer  [b940a8ce] EXACT MATCH (2691 bytes)
```

This means **`dev-agents/roles/*.md` is now the source of truth for agent instructions**, both for SSH-push dispatch (legacy) and Paperclip heartbeat (new). One file, two consumers.

**Rule:** Always specify "verbatim copy from <path>" rather than "use the spirit of <path>" when wiring agent instructions. The CEO followed it byte-for-byte.

### 6. CEO discipline holds when instructed correctly

Across OLY-1, OLY-2, and the cleanup follow-up, the CEO's final reports consistently included:

> Across OLY-2 and this follow-up wake: every action was a hire, a delegation, an org-chart correction, or a comment. **Zero IC work, zero product code written by the CEO.**

This wasn't us trusting the agent — it was us reading the activity log and confirming it. The discipline isn't magic; it's the auto-generated `AGENTS.md` doing its job ("MUST delegate... Do NOT write code yourself"), combined with our task instructions reinforcing it ("hire and delegate; surface blockers; do not paraphrase").

**Rule:** CEO behavior is governed by its `AGENTS.md`. If the CEO is doing IC work, fix the instructions, don't blame the agent.

### 7. Budget caps are runaway guardrails, not real spend control on this setup

Our agents use `claude_local` adapter → Claude Code OAuth → Claude Pro Max subscription (flat-rate). Marginal cost is effectively zero up to Pro Max rate limits. Paperclip's `budgetMonthlyCents` tracks what the spend *would* be at API rates, regardless of actual money flow.

So the cap's real purpose is:
- **Circuit breaker** against runaway loops (CEO → CTO → infinite delegate cycle)
- **Discipline signal** that forces a deliberate "raise the cap" decision

For Olympus at Alpha, recommended cap: **€100/mo** (10000 cents). Catches a runaway loop within a day; doesn't impede normal work; forces a check-in if usage grows.

See `PAPERCLIP.md` § 6.1 for the full breakdown across MAU tiers.

## Open questions for week 2

- **Cleanup hooks**: How do we wire post-task cleanup (delete worktree, prune branch) so we don't repeat the 76-abandoned-worktree pattern from olympus-platform's `dispatch.sh` era? Probably a Paperclip Routine or a per-task `cleanup` step.
- **CTO bottleneck**: All 7 engineers report to one CTO. At what task throughput does the CTO become the limiting factor? Should we add a Tech Lead layer or flatten?
- **API-key vs OAuth split**: Should production-bound work (Olympus's `apps/api/` Anthropic Go SDK calls) and development-bound work (Paperclip agents) share a budget, or be tracked separately?
- **Multi-company isolation**: Once Rios Operator is registered as company #2, verify that an agent in Olympus genuinely cannot read Rios Operator's secrets / instructions. Pen-test before relying on tenancy boundary for real money flow.
- **Skills attachment**: The Orchestrator has `paperclip-create-agent` and `para-memory-files` skills. Do specialist engineers need additional skills attached (e.g., `gh-cli`, `git`, project-specific MCP)? When does adding a skill help vs clutter?

## TL;DR for any future "Day 1 of company N" run

1. Install Paperclip — `make paperclip-up` (idempotent)
2. Create the company in the UI (NOT via API for the first one — UI handles auto-bootstrap better)
3. Ship a read-only first task to validate the heartbeat loop
4. **Before declaring success, check the org chart** — if it's a single-agent company, you're not done
5. Dispatch a hire-team task with explicit role-file-verbatim instructions
6. Verify all `AGENTS.md` byte-for-byte with `cmp`
7. Delete any duplicates (CEO will surface them in its final report)
8. Set budget cap to a runaway-guardrail value (€100/mo for Alpha)
9. Then ship a real implementation task
